# План-чеклист итеративной реализации frontend

## Назначение

Документ фиксирует порядок реализации клиентского мобильного приложения "Шеф-стол" на основе ТЗ, API-контракта и текущей информации о демонстрационном mock-backend.

Frontend должен подтвердить клиентские сценарии MVP: просмотр расписания, карточку класса, создание брони, просмотр и изменение своих броней, отмену, уведомления об отмене студией и оценку шефа.

## Источники истины

- `api/openapi.yaml` — endpoint-ы, схемы, статусы ответов и `errorCode`.
- `TZ_application/03-business-logic.md` — бизнес-правила приложения.
- `TZ_application/04-api-contract-map.md` — связь экранов и API.
- `TZ_application/07-technology-stack.md` — утверждённый frontend-стек.
- `TZ_application/08-feature-list.md` — фичи MVP и приоритеты.
- `design/SCR-*.md` — экранные сценарии.
- `design/SEQ-booking-creation.md` — последовательность создания брони.
- `mock_backend/ARCHITECTURE.md` — архитектура демонстрационного backend и ограничения.
- `mock_backend/IMPLEMENTATION_CHECKLIST.md` — порядок реализации и сценарии mock-backend.
- `logs/TASK-20260705-mock-backend-implementation.md` — текущее состояние реализации mock-backend и ограничения проверки.

## Границы frontend MVP

### Входит

- Expo + React Native + TypeScript.
- Expo Router для навигации.
- TanStack Query для серверного состояния.
- Zustand для сессии, черновиков форм, фильтров и UI-состояний.
- React Hook Form + Zod для форм и валидации.
- Expo SecureStore для access token.
- API-клиент по `api/openapi.yaml`.
- Переключаемый источник данных: mock-backend HTTP API и при необходимости локальный mock-слой для разработки без запущенного backend.
- Обработка обязательных API-ошибок из ТЗ.
- Минимальные тесты ключевых сценариев.

### Не входит

- Реальная онлайн-оплата.
- Эквайринг, платёжные формы, фискализация и автоматические возвраты.
- Админка, интерфейс шефа и управление расписанием.
- Production push/SMS.
- Production auth, OAuth, refresh-token flow и RBAC.
- Создание API/БД/seed для адреса студии.
- Endpoint `/studio-info`, `StudioInfo`, таблица `studios` или любая программная модель адреса студии.
- Маркетинговые фильтры расписания кроме фильтра дат.
- Полноценная CRM, лояльность, поиск, скидки и email/мессенджеры.

## Архитектурное решение frontend

```text
frontend/
  app/
    _layout.tsx
    index.tsx
    auth/
    schedule/
    booking/
    account/
    legal/
  src/
    api/
      client.ts
      errors.ts
      types.ts
      mocks/
    features/
      auth/
      schedule/
      class-details/
      booking/
      account/
      notifications/
      reviews/
      legal/
    shared/
      ui/
      lib/
      config/
      validation/
```

Принципы:

- UI не считает backend ненужным: подтверждённая бронь появляется только после успешного ответа API.
- После мутаций UI использует `Booking`, `BookingMutationResponse` и `ScheduleSlot` из ответа API.
- Предварительный расчёт суммы на клиенте нужен для UX и `expectedTotalAmount`, но финальная сумма берётся из ответа backend.
- Доступность мест, статусы слотов, вместимость и стоимость проката не хардкодятся, если эти данные пришли из API.
- Адрес студии допускается только как статический текст макета, если он есть в дизайне. Он не попадает в API-клиент, типы, mock-data и состояние приложения.

## Переменные окружения

- `EXPO_PUBLIC_API_BASE_URL` — базовый URL mock-backend.
- `EXPO_PUBLIC_API_MODE` — `http` или `local_mock`.
- `EXPO_PUBLIC_DEMO_PHONE` — демонстрационный телефон, если нужен для стенда.
- `EXPO_PUBLIC_DEMO_CODE` — демонстрационный код только для учебного стенда, не production-секрет.

Каждая итерация должна завершаться рабочим состоянием, которое можно запустить и проверить. Если mock-backend временно не собирается или не запущен, допускается локальный mock-слой с теми же схемами и ошибками, но он не должен расходиться с `api/openapi.yaml`.

## Риски

- Mock-backend в текущем логе реализации ещё не подтверждён сборкой и тестами из-за отсутствия Go и недоступного Docker daemon.
- Локальный mock-слой может разойтись с HTTP mock-backend, если не сверять его с `api/openapi.yaml`.
- Есть риск случайно реализовать онлайн-оплату вместо заглушки. Это выходит за MVP.
- Есть риск вернуть адрес студии в программный контур. Это уже зафиксированный баг-класс, его нельзя повторять.
- UI-проверка правила 24 часов нужна для UX, но окончательное решение всегда за backend.
