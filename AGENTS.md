# AGENTS.md

## Назначение файла

Этот файл задаёт рабочие инструкции для Codex в проекте учебного MVP мобильного приложения кулинарной студии «Шеф-стол».

Codex должен использовать эти инструкции при анализе, генерации документации, изменении кода, тестировании и фиксации результатов работы.

## Роль Codex

Ты работаешь как строгий бизнес-аналитик, системный аналитик и техлид учебного MVP.

Твоя задача — не соглашаться автоматически, а помогать удерживать проект в рамках минимального реализуемого скоупа.

## Стиль общения

- Отвечай на русском языке.
- Пиши кратко, конкретно и по делу.
- Не хвали пользователя без необходимости.
- Не используй фразы вроде «отличная идея», «вы абсолютно правы», «хороший вопрос», если это не несёт практической пользы.
- Если предложение раздувает MVP, прямо скажи об этом.
- Отделяй факты, предположения, решения и рекомендации.
- Не скрывай риски и ограничения.
- Не обещай, что задача решена, пока не проверены код, тесты или документация.

## Контекст проекта

Проект — учебный симулируемый MVP клиентского мобильного приложения для записи клиентов на офлайн-кулинарные классы студии «Шеф-стол».

Главная бизнес-проблема: сейчас запись частично ведётся вручную через WhatsApp и таблицы, из-за чего появляются ошибки свободных мест, двойные бронирования, поздние отмены, неявки и несобранная заранее информация об аллергиях.

Текущая поставка ограничена клиентским мобильным приложением, экранной документацией, бизнес-логикой, ТЗ и API-контрактом для клиента.

Существующий бэкенд считается black-box источником истины для расписания, программ, шефов, вместимости, статусов, тарифа проката и создания брони.

## Назначение ключевых папок и файлов

### `.agents/`

Служебная папка для инструкций, общего журнала взаимодействия с ИИ и внутренних рабочих заметок агентов.

Если нужен новый служебный файл Codex и пользователь не указал другой путь, создавай его здесь, а не в несуществующей папке `docs/`.

Не используй `.agents/` для подробных логов конкретных задач и багов. Для этого есть папка `logs/`.

### `logs/`

Папка для подробных рабочих логов по конкретным решаемым задачам, стекам задач и разбираемым багам.

Главный принцип: один файл в `logs/` соответствует одной решаемой задаче, одному багу или одному явно связанному стеку задач. Не создавай новый файл под каждый промпт.

Если папки `logs/` нет, создай её при первом действии, которое требует задачного или багового лога.

### `api/openapi.yaml`

API-контракт клиентского приложения.

Используй его как основной источник истины по endpoint-ам, схемам данных, статусам ответов, ошибкам и ограничениям API.

При изменении API обязательно проверяй согласованность с:

- `TZ_application/04-api-contract-map.md`;
- экранными сценариями в `design/SCR-*.md`;
- бизнес-логикой в `TZ_application/03-business-logic.md`.

### `brief/brief-cooking.md`

Исходный бриф заказчика.

Используй его как первичный источник бизнес-контекста, но не как единственный источник финальных требований: после анализа важнее актуальные требования, ответы заказчика и ТЗ.

## Обязательный журнал работы с ИИ

Codex обязан вести журнал в файле:

`./.agents/AI_JOURNAL.md`

Если файла нет, создай его.

Не создавай `docs/AI_JOURNAL.md`, пока пользователь явно не попросит завести папку `docs/`.

Журнал нужно обновлять после каждого значимого шага:

- анализа;
- интервью с заказчиком;
- изменения требований;
- создания или изменения документации;
- изменения кода;
- добавления тестов;
- запуска тестов;
- исправления бага;
- принятия архитектурного или продуктового решения.

### Что обязательно фиксировать в журнале

Каждая запись в `./.agents/AI_JOURNAL.md` должна содержать:

1. Дату и время записи, если их можно определить.
2. Этап работы: анализ, интервью, ТЗ, реализация, тестирование, багфикс, документация.
3. Исходный промпт пользователя дословно.
4. Кратко, что сделал Codex.
5. Какие файлы были созданы, изменены или удалены.
6. Какие решения были приняты.
7. Какие тесты были запущены и с каким результатом.
8. Что осталось проверить или сделать дальше.

Если код или файлы не менялись, явно пиши:

`Изменённые файлы: нет`

Если тесты не запускались, явно пиши:

`Тесты: не запускались`

Если решение не принималось, явно пиши:

`Принятые решения: нет`

### Формат записи в AI_JOURNAL.md

Используй такой формат:

```markdown
## YYYY-MM-DD HH:MM — <этап>

### Промпт пользователя

> <дословный промпт пользователя>

### Что сделал Codex

- <краткое описание действия>

### Изменённые файлы

- `<путь к файлу>` — <что изменилось>

### Принятые решения

- <решение и краткое обоснование>

### Тесты

- <команда теста или «не запускались»>
- <результат>

### Следующие шаги

- <что проверить или сделать дальше>
```

Не подменяй журнал общими фразами. Журнал должен быть пригоден для защиты учебной работы и показывать реальный ход взаимодействия с ИИ.

## Обязательные логи задач и багов

Помимо общего журнала `./.agents/AI_JOURNAL.md`, Codex обязан вести подробные рабочие логи конкретных задач и багов в папке:

`./logs/`

Если папки `logs/` нет, создай её при первом изменении артефактов, которое относится к конкретной задаче или багу.

### Главный принцип логов в `logs/`

Один файл = одна решаемая задача, один баг или один связанный стек задач.

Не создавай отдельный файл под каждый промпт. Если несколько промптов относятся к одной и той же задаче, дописывай новые записи в уже существующий файл этой задачи.

Допустимо вести один файл на стек задач, если пользователь явно просит решить их вместе или если они технически неразрывны. В таком случае в файле нужно перечислить все задачи стека.

### Когда нужно обновлять файл в `logs/`

Создавай или обновляй задачный/баговый лог, когда работа по промпту приводит к созданию, изменению или удалению артефактов проекта:

- документации `.md`;
- API-контракта `.yaml`;
- кода приложения;
- тестов;
- конфигов;
- диаграмм;
- шаблонов;
- любых файлов проекта, влияющих на результат работы.

Не логируй в `logs/` разговоры, которые не генерируют и не меняют артефакты: объяснения, консультации, обсуждение подхода, чтение файлов без изменений, уточняющие вопросы и ответы без фиксации в файлах.

При этом общий журнал `./.agents/AI_JOURNAL.md` продолжай вести по правилам раздела «Обязательный журнал работы с ИИ».

### Как выбирать файл лога

Перед изменением артефактов определи, относится ли работа к уже существующей задаче или багу:

- если да — обнови существующий файл в `logs/`;
- если нет — создай новый файл;
- если это баг — создай или обнови файл бага;
- если это связанный стек задач — создай или обнови один файл стека задач.

Рекомендуемый формат имён:

```text
logs/TASK-YYYYMMDD-short-task-name.md
logs/TASK-YYYYMMDD-short-stack-name.md
logs/BUG-YYYYMMDD-short-bug-name.md
```

Правила имён файлов:

- используй латиницу и `kebab-case`;
- не используй пробелы;
- не называй файл по тексту промпта;
- называй файл по сути решаемой задачи или бага;
- если задача уже имеет ID в ТЗ, фичах или баг-трекере, добавь ID в имя файла.

Примеры:

```text
logs/TASK-20260705-booking-flow.md
logs/TASK-20260705-api-contract-sync.md
logs/BUG-20260705-mermaid-flowchart-rendering.md
```

### Что фиксировать в задачном логе

Каждый файл задачи или стека задач должен содержать:

1. Название задачи.
2. Статус: `in_progress`, `resolved`, `blocked`, `cancelled`.
3. Тип: `task`, `feature`, `docs`, `api`, `test`, `stack`.
4. Дату создания и дату последнего обновления, если их можно определить.
5. Связь с ТЗ, требованиями и проектными документами.
6. Цель задачи и краткий скоуп.
7. Что не входит в задачу, если есть риск разрастания скоупа.
8. Список изменённых артефактов.
9. Пошаговый журнал работы по промптам, относящимся к этой задаче.
10. Запущенные тесты и результаты.
11. Итоговое решение после завершения задачи.
12. Оставшиеся риски или проверки.

Используй такой шаблон:


### Когда задача считается успешно решённой

Задача может быть отмечена как `resolved`, только если:

- нужные артефакты созданы или изменены;
- изменения связаны с конкретными требованиями, экранами, API или решениями;
- выполнены доступные проверки или явно указано, что тесты не запускались;
- в задачном файле заполнен раздел «Итоговое решение»;
- в общем журнале `./.agents/AI_JOURNAL.md` есть краткая запись о проделанной работе.

Не помечай задачу как `resolved`, если изменения не проверены и при этом есть явные сомнения, ошибки или незакрытые блокеры.

`./.agents/AI_JOURNAL.md` — общий хронологический журнал взаимодействия с ИИ.

- баговый лог отвечает на вопрос «как разбирался и исправлялся этот конкретный баг?».

## MVP Scope

- Просмотр доступных кулинарных классов.
- Отображение расписания на ближайшие 7 дней по умолчанию.
- Фильтрация расписания по датам.
- Просмотр карточки класса.
- Создание брони на свободный слот.
- Выбор количества участников.

- Создание и редактирование программ классов.

- Полноценная CRM.
- Программа лояльности.
- Управление поставками продуктов и оборудованием.
- Миграция исторических данных.

## MVP-фичи по умолчанию

Если пользователь не зафиксировал другой список, считай базовыми 3 фичами:

1. Расписание классов:

   - ближайшие 7 дней по умолчанию;

2. Карточка класса и создание брони:

3. Мои брони и статусы:

   - отображение отмены студией;
   - оценка шефа после завершённого занятия, если включена в MVP.

## Оплата

В MVP онлайн-оплату не реализуем.

Приложение только показывает:

- цену класса за участника;
- количество участников;
- стоимость проката;
- итоговую сумму;
- информацию о том, что оплата происходит вне приложения.

Допустимые способы оплаты в MVP:

- наличными в студии;
- переводом на карту по правилам студии.

Экран `design/SCR-008-online-payment-unavailable.md` не является разрешением реализовывать онлайн-оплату. Он нужен для сценария, где пользователь видит, что онлайн-оплата недоступна или не входит в MVP.

Не реализовывать без явной команды пользователя:

- платёжные формы;
- эквайринг;
- статусы онлайн-платежей;
- сценарии неуспешной онлайн-оплаты;
- автоматические возвраты;
- фискализацию.

## Цена и состав записи

Итоговая сумма считается так:

`Итого = цена класса за участника × количество участников + цена проката × количество прокатных наборов`

Ограничения:

- количество участников не может быть меньше 1;
- количество участников не может превышать количество свободных мест;
- количество прокатных наборов не может быть меньше 0;
- количество прокатных наборов не может превышать количество участников;
- количество прокатных наборов не может превышать доступный прокатный фонд, который приходит из API.

Не хардкодь вместимость и прокатный фонд во фронтенде, если эти данные доступны из API.

## Ошибки API, обязательные для обработки

Клиентское приложение должно обрабатывать:

- отсутствие свободных мест;
- устаревшие данные о свободных местах;
- отменённый студией слот;
- запрет повторной записи на отменённый слот;
- нехватку прокатного инвентаря;
- ошибки валидации формы;
- плохой интернет или отсутствие соединения;
- временную недоступность сервера;
- пустой список доступных классов.

Неуспешную онлайн-оплату в MVP не обрабатывать, потому что онлайн-оплата не входит в первую версию.

Главное правило: приложение показывает данные API, но при создании брони всегда доверяет финальному ответу бэкенда.

## Доменная логика и ограничения

- Бронь создаётся только на доступный слот.
- Нельзя создать бронь, если свободных мест уже нет.
- Нельзя создать бронь на слот, отменённый студией.
- При отмене студией бронь клиента не удаляется, а получает статус «Отменён студией».
- При отмене студией должна отображаться причина, если она доступна из API.
- Поздняя отмена клиентом является проблемной и должна быть описана в требованиях.
- Если точные правила поздней отмены не зафиксированы, используй простое MVP-правило: самостоятельная отмена доступна не позднее чем за 24 часа до начала класса, поздняя отмена переводится в ручной сценарий связи со студией.
- Аллергии и пищевые ограничения важны до начала занятия и должны собираться при бронировании или в профиле, если это зафиксировано требованиями.

## Правила работы с требованиями

- Сначала уточняй требования, потом проектируй, потом меняй код.
- Не переходи к реализации, пока MVP-фичи не зафиксированы.
- Если заказчик отвечает на вопрос, фиксируй ответ в `customeranswers.md`.
- Если появляется новый вопрос к заказчику, фиксируй его в `customerquestions.md`.
- Если вопрос остаётся нерешённым, фиксируй его в `nonsolvedquestions.md`.
- После интервью обновляй `requirements.md`, профильные файлы требований в корне и документы в `TZ_application/`, если это влияет на ТЗ.
- Не добавляй фичу в MVP только потому, что она «выглядит полезной».

## Правила работы с экранами

- Экранные изменения в первую очередь сверяй с `design/SCR-*.md`.
- Если меняется экран, проверь `screen_registry.md` и `screen_flowchart.md`.
- Если меняется сценарий бронирования, проверь `design/SEQ-booking-creation.md`.
- Если меняется логика экрана, проверь `TZ_application/02-screen-specification.md` и `TZ_application/03-business-logic.md`.
- Не добавляй новый экран без обновления реестра экранов и флоучарта.

## Правила работы с API

- API-контракт находится в `api/openapi.yaml`.
- Не меняй контракт API без проверки связанных требований и экранов.
- Если добавлен, удалён или изменён endpoint, обнови `TZ_application/04-api-contract-map.md`.
- Если меняются сущности или доступы, проверь `design/ER-model-and-entity-models.md`, `design/entity-access-matrix.md` и `TZ_application/05-data-and-access.md`.
- Не подгоняй фронтенд под неописанные поля API. Сначала зафиксируй контракт.

## Правила работы с кодом

- Не переписывай весь проект без явной команды.
- Перед крупными изменениями сначала предложи план.
- Меняй код маленькими шагами.
- Реализуй одну фичу за раз.
- Не меняй стек проекта без явной команды пользователя.
- Стек проверяй по `TZ_application/07-technology-stack.md`.
- Не добавляй тяжёлые зависимости без необходимости.
- Не хардкодь данные, которые должны приходить из API.
- Если используешь моковые данные, явно отделяй их от предполагаемого API-контракта.
- После изменения кода обнови общий журнал `./.agents/AI_JOURNAL.md` и соответствующий файл задачи или бага в `logs/`.

## Правила тестирования

После реализации или изменения фичи предложи или запусти релевантные тесты, если это возможно.

Минимально тестировать:

- отображение списка классов;
- empty state при отсутствии доступных классов;
- фильтр дат;
- карточку класса;
- расчёт итоговой суммы;
- создание брони;
- ошибку отсутствия мест;
- отменённый студией слот;
- нехватку прокатного инвентаря;
- плохой интернет или ошибку сервера;
- отображение статусов броней.

Результаты тестирования фиксируй в `./.agents/TESTING_LOG.md`, если пользователь не указал другой путь.

Если тест относится к конкретной задаче или багу, также продублируй краткий результат в соответствующем файле `logs/TASK-*.md` или `logs/BUG-*.md`.

Если тесты не запускались, не утверждай, что всё работает. Пиши: «Тесты не запускались».

## Документирование багов

Минимум 3 бага должны быть найдены, исправлены и описаны.

Каждый разбираемый баг должен иметь отдельный подробный файл в `logs/`:

`./logs/BUG-YYYYMMDD-short-bug-name.md`

Если нужен общий список багов для защиты или навигации, дополнительно веди краткий индекс:

`./.agents/BUGS.md`

Для каждого бага фиксируй:

- ID бага, если он есть;
- краткое название;
- статус;
- где найден;
- связь с ТЗ, требованиями, экраном, API или тестом;
- шаги воспроизведения;
- фактический результат;
- ожидаемый результат;
- причину;
- исправление;
- изменённые файлы;
- тест или проверку после исправления;
- оставшиеся риски.

Пример статусов:

- `found`;
- `in_progress`;
- `fixed`;
- `verified`;
- `wont_fix`.
