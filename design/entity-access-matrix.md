# Сущности и права доступа приложения

## Назначение документа

Документ фиксирует права клиентского мобильного приложения на чтение и изменение сущностей. Это не матрица прав внутреннего бэкенда и не описание админки: шефские и административные интерфейсы существуют вне текущей поставки.

Главный принцип: мобильное приложение работает только в клиентском сценарии. Оно читает публичные данные расписания и классов, а персональные данные изменяет только от имени авторизованного клиента и только через API.

## Роли и системные участники

| Участник | Входит в приложение | Назначение |
|---|---|---|
| Гость | Да | Неавторизованный пользователь, который может открыть расписание и карточку класса. |
| Авторизованный клиент | Да | Пользователь, вошедший по телефону и SMS-коду/временному паролю. Может создавать и изменять только свои брони. |
| Мобильное приложение | Да | Клиент API; хранит только временные черновики форм и отображает подтверждённые данные API. |
| Существующий бэкенд | Нет, внешний системный участник | Источник истины для расписания, доступности, броней, статусов, уведомлений и проверок прав. |
| Шеф | Нет | Роль есть в домене, но интерфейс шефа вне скоупа приложения. |
| Владелец / администратор | Нет | Управляет расписанием, программами, шефами и административными отменами во внешней админке. |

## Обозначения операций

| Обозначение | Значение |
|---|---|
| `R` | чтение через API |
| `C` | создание через API |
| `U` | изменение через API |
| `D` | удаление; для бизнес-сущностей приложения не используется, вместо удаления применяются статусы отмены |
| `-` | действие недоступно приложению |

## Правила доступа к персональным данным

1. Номер телефона клиента доступен только авторизованному клиенту в рамках его сессии и бэкенду.
2. Имена участников, бинарный признак аллергии и комментарий об аллергии доступны только в контексте конкретной брони клиента.
3. Аллергии не сохраняются в профиле клиента и не переиспользуются между бронями.
4. Приложение не должно показывать данные чужих броней, участников, отзывов и уведомлений.
5. При ошибке API приложение не создаёт локальную подтверждённую бронь и не считает локальный черновик источником истины.

## Правила операций

### Авторизация

- Гость может запросить вход по телефону.
- После успешного подтверждения кода бэкенд возвращает сессию клиента.
- Если аккаунта ещё нет, бэкенд создаёт его автоматически.

### Просмотр расписания

- Гость и авторизованный клиент могут читать расписание, карточку класса, шефа, свободные места, цену места и условия проката.
- Приложение не может создавать или редактировать расписание, программы, шефов и административные отмены.
- Доступность слота всегда считается по данным API.

### Создание брони

- Создание брони доступно только авторизованному клиенту.
- Клиент может создать бронь только на доступный слот.
- Бронь считается созданной только после успешного ответа API.
- При отказе API из-за отсутствия мест, устаревших данных, отменённого слота или ошибок валидации приложение показывает ошибку и обновляет данные слота.

### Добавление участников

- Добавление участников доступно только авторизованному клиенту и только в его собственной предстоящей брони.
- Добавление возможно только при наличии мест и после подтверждения API.
- Данные каждого участника включают имя, выбор аллергии `есть/нет`, комментарий при наличии аллергии и выбор инвентаря `own/rental`.

### Отмена брони или участников

- Клиент может отменить только свою бронь или участников своей групповой брони.
- Отмена доступна не позднее чем за 24 часа до начала класса.
- Удаления записи из модели нет: бэкенд меняет статус брони или участника.
- Освобождение мест отображается только после обновления данных из API.

### Отмена класса студией

- Клиентское приложение не инициирует административную отмену класса.
- При отмене класса во внешней админке бэкенд возвращает статус `cancelled_by_studio`.
- `cancellationReason` опционален: если причина не пришла, приложение показывает нейтральный текст о том, что класс отменён студией.
- Повторная запись на отменённый студией слот запрещена.

### Оценка шефа

- Оценка доступна только авторизованному клиенту по его завершённой брони.
- Оценка от 1 до 5 обязательна для отправки, текстовый отзыв необязателен.
- Повторная отправка оценки по той же брони должна обрабатываться как состояние “оценка уже отправлена”.

## Границы ответственности

| Зона | Кто отвечает |
|---|---|
| Проверка, что бронь принадлежит клиенту | Бэкенд |
| Запрет двойного бронирования | Бэкенд |
| Проверка правила 24 часов | Бэкенд, приложение дополнительно предупреждает в UI |
| Расчёт финальной доступности мест | Бэкенд |
| Предварительная валидация формы | Приложение |
| Хранение черновика формы | Приложение локально |
| Подтверждение бизнес-состояния | Только API |
| Административное управление расписанием | Внешняя админка |

## Инварианты доступа

1. В приложении нет ролей шефа, владельца и администратора.
2. Приложение не предоставляет операции удаления бизнес-сущностей.
3. Любое изменение брони, участников, выбора оплаты или оценки шефа требует авторизованного клиента.
4. Любое изменение, влияющее на места, считается успешным только после подтверждения API.
5. Персональные данные участника доступны только через его бронь и не становятся частью постоянного профиля клиента.
6. Публичные данные расписания можно читать без авторизации, но любые действия с бронью требуют входа.
7. Адрес единственной студии не входит в матрицу доступа: это статический UI/UX-текст макета, не сущность, не API-данные, не локальное состояние и не объект внешней админки.
