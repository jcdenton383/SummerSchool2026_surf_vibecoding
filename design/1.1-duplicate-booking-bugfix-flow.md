# Версия 1.1 — исправление повторной записи на один класс

## Тип

Bugfix.

## Проблема

Клиент мог создать несколько активных броней на один и тот же слот. Это искажает вместимость, список записей и расчёт группы.

## Новое правило

Для клиента и слота допускается только одна активная бронь. Если активная бронь уже существует, новая бронь не создаётся.

## Пользовательский сценарий

1. Клиент авторизован.
2. Клиент уже имеет активную бронь на слот.
3. Клиент открывает этот слот из расписания.
4. Приложение явно сообщает, что клиент уже записан.
5. Приложение предлагает добавить дополнительных участников в существующую группу.
6. Стоимость считается только за новых участников и выбранный для них прокат.

## Backend-правило

Перед созданием брони backend проверяет активные брони по `clientId + slotId`.

Активные статусы:

- `confirmed`;
- `offline_payment_pending`.

При найденной активной брони backend возвращает управляемую ошибку `already_exists`.

## Проверка

- Повторный `POST /bookings` на тот же слот для того же клиента должен вернуть ошибку.
- `POST /bookings/{bookingId}/participants` должен оставаться доступным для редактируемой активной брони при наличии мест.
