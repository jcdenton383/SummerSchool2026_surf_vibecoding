# Требования к MVP клиентского мобильного приложения «Шеф-стол»

## 1. Цель MVP

MVP должен дать клиентам возможность самостоятельно записываться на кулинарные классы студии «Шеф-стол» через кроссплатформенное мобильное приложение для Android и iOS. Приложение должно снизить ручную запись через WhatsApp и таблицы, показать клиенту доступные классы, оформить бронь, собрать данные участников, учесть инвентарь, аллергию, отмены и базовые уведомления.

Существующий бэкенд, админка владельца и интерфейсы шефов уже реализованы. Клиентское приложение работает через API и не создаёт расписание, программы или шефов самостоятельно.

## 2. Состав MVP

В MVP входят:

1. Авторизация по телефону через SMS-код.
2. Автоматическое создание аккаунта клиента по номеру телефона.
3. Кроссплатформенное мобильное приложение для Android и iOS.
4. Просмотр классов на ближайшие 7 дней от текущего момента по умолчанию.
5. Фильтр даты для просмотра более длинного периода расписания.
6. Empty state «Пока нет доступных классов», если расписания на ближайшие дни нет.
7. Наличие нескольких типов классов с направлением, сложностью и привязанным шефом.
8. Просмотр свободных мест с учётом разной максимальной вместимости классов.
9. Карточка класса с названием, длительностью, сложностью, блюдами, именем шефа и необходимым инвентарём.
10. Запись одного клиента или группы участников.
11. Добавление участников в уже созданную запись.
12. Выбор собственного инвентаря или платного проката для каждого участника.
13. Отображение стоимости мест, проката и итоговой суммы.
14. Выбор оплаты оффлайн или онлайн-заглушки.
15. Отмена записи не позднее чем за 24 часа до класса.
16. Селективная отмена участников в групповой записи.
17. Обязательный сбор информации об аллергиях по каждому участнику.
18. Push-напоминание за 1 день до класса.
19. SMS дополнительно к push при отмене класса студией.
20. Оценка шефа после завершённого класса по пятизвёздочной шкале.
21. Необязательный текстовый отзыв.
22. Условия отмены и политика обработки данных.
23. Отображение адреса единственной студии на баннере с названием студии.

Не входят в MVP:

1. Управление расписанием, программами и шефами.
2. Интерфейсы администратора, владельца и шефа.
3. Реальная онлайн-оплата.
4. Реальный автоматический возврат денег.
5. Лояльность, бонусы, скидки и статусы постоянных клиентов.
6. Поиск и маркетинговые фильтры по каталогу, кроме фильтра даты.
7. Email-уведомления и уведомления через мессенджеры.

## 3. Пользовательские роли

### Клиент

Основная роль MVP. Клиент просматривает классы, создаёт аккаунт по телефону, записывает себя и других участников, выбирает инвентарь, указывает аллергии, видит свои предстоящие и прошедшие классы, отменяет записи в допустимый срок и оценивает шефа.

### Администратор / владелец

Не является пользователем текущего приложения. Действует через существующую инфраструктуру. Может отменить класс по форс-мажору во внешней системе; клиентское приложение только получает статус отмены и показывает его клиенту.

### Шеф

Не является пользователем текущего приложения. Отображается в карточке класса и получает оценки от клиентов.

## 4. Пользовательские сценарии

### Регистрация и вход

1. Клиент устанавливает приложение.
2. Вводит номер телефона.
3. Получает SMS-код или временный пароль.
4. Вводит код в приложении.
5. Система автоматически создаёт аккаунт или открывает существующий.
6. После входа клиент может создать постоянный пароль или продолжать входить по SMS.

### Запись одного участника

1. Клиент открывает приложение.
2. Видит список доступных классов.
3. Выбирает понравившийся класс.
4. Открывает расширенную карточку класса.
5. Видит название, длительность, сложность, блюда, шефа, нужный инвентарь, свободные места и стоимость.
6. Выбирает запись одного участника.
7. Указывает имя и телефон.
8. Выбирает, есть ли аллергии.
9. Если аллергии есть, вводит краткий комментарий.
10. Выбирает собственный инвентарь или прокат.
11. Видит состав записи и итоговую сумму.
12. Выбирает способ оплаты: оффлайн или онлайн-заглушку.
13. Система отправляет запрос на создание брони в API.
14. При успешном ответе API место помечается занятым, а запись появляется в аккаунте клиента.

### Запись группы

1. Клиент выбирает класс.
2. Указывает себя как основного участника.
3. Добавляет дополнительных участников.
4. Для каждого дополнительного участника указывает имя.
5. Для каждого участника явно выбирает «аллергии есть / нет».
6. При наличии аллергии указывает комментарий.
7. Для каждого участника выбирает свой инвентарь или прокат.
8. Система проверяет через API возможность забронировать нужное количество мест.
9. Если мест достаточно, создаётся групповая бронь.
10. Если мест недостаточно, приложение показывает понятное сообщение и обновляет данные слота.

### Добавление участников в существующую запись

1. Клиент открывает аккаунт.
2. Выбирает предстоящий класс.
3. Нажимает добавление участников.
4. Вводит N дополнительных участников и их данные.
5. Для каждого нового участника явно выбирает «аллергии есть / нет».
6. При наличии аллергии указывает комментарий.
7. Выбирает инвентарь для каждого.
8. Видит доплату за добавленные места и прокат.
9. Выбирает метод оплаты.
10. Система через API проверяет свободные места.
11. Если мест хватает, N мест добавляются к записи.
12. Если мест не хватает, приложение сообщает, что добавить указанное количество участников нельзя.

### Отмена записи

1. Клиент открывает аккаунт.
2. Выбирает предстоящий класс.
3. Если до класса осталось не менее 24 часов, может отменить запись.
4. Для групповой записи клиент может выбрать M участников для отмены.
5. Система показывает заглушку возврата денег.
6. После подтверждения отмены приложение отправляет запрос в API.
7. API освобождает M мест.
8. Освобождённые места отображаются другим клиентам как свободные.

Если до класса меньше 24 часов:

1. Клиент открывает предстоящий класс.
2. Пытается отменить бронь.
3. Приложение блокирует действие.
4. Клиент видит сообщение, что отмена менее чем за 24 часа до класса невозможна.

Правило невозможности отмены менее чем за 24 часа должно быть показано ещё при оформлении заказа.

### Оценка шефа

1. После завершения класса клиент открывает приложение.
2. Открывает аккаунт.
3. Переходит в недавние классы.
4. Выбирает завершённый класс.
5. Ставит оценку шефу по шкале 1-5 звёзд.
6. При желании оставляет текстовый отзыв.
7. Оценка сохраняется и влияет на рейтинг шефа, видимый клиентам в расширенной карточке класса.

### Форс-мажорная отмена класса

1. Администратор/заказчик отменяет класс во внешней системе.
2. Бэкенд передаёт клиентскому приложению статус отмены.
3. Всем записанным клиентам приходит push-уведомление.
4. Всем записанным клиентам приходит SMS с информацией об отмене.
5. В приложении у класса отображается статус отмены.
6. В приложении показывается яркая инструкция о возврате денег.
7. Инструкция по возврату реализуется как заглушка.

Административное действие отмены класса не входит в клиентское приложение.

## 5. Данные и поля

### Класс

1. Название.
2. Направление.
3. Сложность.
4. Длительность.
5. Блюда.
6. Имя шефа.
7. Рейтинг шефа.
8. Необходимый инвентарь.
9. Дата и время.
10. Максимальная вместимость.
11. Количество свободных мест.
12. Стоимость места.
13. Стоимость проката инвентаря.

### Основной клиент

1. Имя.
2. Номер телефона.
3. Статус аллергий по конкретной записи.
4. Комментарий по аллергии, если аллергия есть.

### Дополнительный участник

1. Имя.
2. Статус аллергий по конкретной записи.
3. Комментарий по аллергии, если аллергия есть.
4. Выбор собственного инвентаря или проката.

Данные об аллергиях не сохраняются как постоянный профиль клиента. Они хранятся на уровне конкретной брони и участника.

## 6. Оплата и статусы

В MVP реальная онлайн-оплата не проводится. Приложение показывает состав записи и итоговую сумму.

Способы оплаты в интерфейсе:

1. Оффлайн-оплата перед началом класса.
2. Онлайн-оплата как заглушка.

При выборе онлайн-оплаты приложение показывает заглушку и предлагает оплатить лично оффлайн перед началом класса. Если клиент соглашается, создаётся бронь со статусом оффлайн-оплаты. Если отказывается, бронь не создаётся. До выбора оффлайн-оплаты после заглушки бронь не создаётся.

Клиентские состояния:

1. Свободно.
2. Ожидает оплаты перед классом, рабочее название ранее: «оплата оффлайн».
3. Забронировано, зарезервировано для будущего сценария реальной онлайн-оплаты.

Финальное название статуса для оффлайн-оплаты нужно уточнить отдельно.

## 7. Уведомления

Push-уведомления в MVP:

1. Напоминание о классе за 1 день до начала.
2. Отмена класса студией.

Дополнительное правило:

1. При отмене класса студией отправляется и push-уведомление, и SMS.
2. Если в заказе класса не выбран прокат инвентаря, push-напоминание за день до класса должно включать фразу «не забудьте инвентарь для класса» или близкую по смыслу.

## 8. API и обработка ошибок

Клиентское приложение не решает самостоятельно, есть ли место в классе. Оно показывает данные API, но при создании или изменении брони всегда доверяет финальному ответу бэкенда.

Приложение должно обрабатывать:

1. Отсутствие мест.
2. Устаревшие данные о свободных местах.
3. Отменённый слот.
4. Запрет повторной записи на отменённый студией слот.
5. Ошибки валидации.
6. Плохой интернет.
7. Временную недоступность сервера.
8. Попытку добавить больше участников, чем доступно свободных мест.
9. Попытку отменить бронь менее чем за 24 часа до класса.

При отказе API приложение должно:

1. Показать понятное сообщение.
2. Не создавать локально неподтверждённую бронь.
3. Обновить данные слота.
4. Дать клиенту возможность вернуться к выбору или изменить состав записи.

API-контракт клиентского приложения должен включать:

1. Получение списка классов и слотов на ближайшие 7 дней по умолчанию и за выбранный диапазон дат при использовании фильтра даты.
2. Получение карточки класса с программой, меню, шефом, рейтингом шефа, датой, временем, длительностью, вместимостью, свободными местами, стоимостью места и тарифом проката.
3. Создание брони одного участника и групповой брони.
4. Добавление участников в существующую предстоящую бронь.
5. Отмену всей брони и отдельных участников с освобождением мест.
6. Получение клиентских статусов брони: подтверждена, ожидает оффлайн-оплаты, отменена клиентом, отменена студией, завершена.
7. Сохранение оценки шефа от 1 до 5 и необязательного текстового отзыва после завершения класса.
8. Типизированные ошибки для отсутствия мест, устаревших данных, отменённого слота, запрета повторной записи, ошибок валидации и временной недоступности сервера.

## 9. Бизнес-ограничения

1. По умолчанию клиенту показываются слоты на ближайшие 7 дней от текущего момента.
2. Более длинный период расписания доступен через фильтр даты.
3. Если расписания на ближайшие дни нет, приложение показывает empty state «Пока нет доступных классов».
4. Вместимость класса зависит от типа/направления/сложности и может отличаться между классами.
5. Нельзя забронировать больше мест, чем доступно по API.
6. Отмена клиентом доступна только не позднее чем за 24 часа до начала класса.
7. После отмены места должны снова стать свободными.
8. Аллергии должны быть явно указаны по каждому участнику.
9. Без ответа по аллергиям запись оформить нельзя.
10. При выборе «аллергии есть» комментарий обязателен.
11. Прокат инвентаря оплачивается отдельно и может иметь разную стоимость для разных классов.
12. Условия отмены и политика обработки данных должны быть доступны клиенту.
13. Адрес единственной студии должен отображаться на баннере с названием студии и не должен передаваться как поле API расписания или карточки класса.

## 10. Критерии приёмки MVP

MVP считается готовым, если выполняются следующие критерии:

1. Клиент может создать аккаунт через телефон и SMS-код.
2. Клиент по умолчанию видит список классов на ближайшие 7 дней от текущего момента.
3. Клиент может выбрать более длинный период через фильтр даты.
4. Если расписания на ближайшие дни нет, клиент видит empty state «Пока нет доступных классов».
5. Клиент может открыть карточку класса.
6. В карточке класса отображаются название, длительность, сложность, блюда, шеф, нужный инвентарь, свободные места и стоимость.
7. Клиент может записать себя одного на класс.
8. Клиент не может оформить запись без явного ответа по аллергиям.
9. Клиент может оформить групповую запись.
10. Клиент может добавить участников в существующую запись.
11. Система не позволяет добавить больше участников, чем есть свободных мест.
12. Клиент может выбрать прокат или собственный инвентарь для каждого участника.
13. Приложение показывает итоговую сумму до подтверждения записи.
14. При выборе онлайн-оплаты показывается заглушка и предлагается оффлайн-оплата.
15. Успешная запись появляется в аккаунте клиента.
16. Клиент может отменить всю запись или отдельных участников не позднее чем за 24 часа до класса.
17. Освобождённые места после отмены снова отображаются как свободные.
18. Клиент не может отменить запись менее чем за 24 часа до класса.
19. Ограничение отмены видно при оформлении заказа и в карточке брони.
20. Клиент может поставить шефу оценку 1-5 звёзд после завершения класса.
21. Текстовый отзыв можно оставить, но он не обязателен.
22. За 1 день до класса клиент получает push-напоминание.
23. Если в заказе класса не выбран прокат инвентаря, push-напоминание содержит фразу «не забудьте инвентарь для класса» или близкую по смыслу.
24. При отмене класса студией клиент получает push и SMS.
25. При отмене класса студией в приложении отображается заметная инструкция по возврату денег.
26. Адрес студии может отображаться на баннере только как статический UI/UX-текст макета; это не функциональное требование к API, БД, seed-данным или mock-backend.
27. Приложение корректно обрабатывает отсутствие мест, устаревшие данные, отменённый слот, ошибки валидации, плохой интернет и недоступность сервера.

## 11. Открытый вопрос

1. Нужно выбрать финальное клиентское название статуса для брони, при которой клиент планирует оплатить лично перед началом класса.
