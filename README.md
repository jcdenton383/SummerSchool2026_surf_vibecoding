# SummerSchool2026_surf_vibecoding
Surf Studio summer school vibecoding project

Цикл "разработки" повторял цикл, описанный на 1 и 2 лекциях Летней Школы, без использования вспомогательных файлов школы.
Бриф: «Кулинарная студия»

На этапе разработки технического задания использовалась бесплатная версия ChatGPT, на этапе проектирования и имплементации использовалась платная версия CHatgpt + Codex, с созданием файла AGENTS.md, регламентирующего поведение агента.

1. Разработка технического задания.
На данном этапе использовались промпты, повторяющие промпты на лекции. Основной целью на этом этапе было проверить, способна ли бесплатная версия ChatGPT справиться с анализом брифа в достаточной мере. Вероятные пробелы в техническом задании были оставлены с целью оценки влияния "hands-off" подхода при составлении документации. При попытке имплементации подхода "открытый вопрос-ответ", предложенный на лекции, нейронная сеть предложила тидбит информации, не представленный в брифе и дополнении к нему: "адрес студии". Эта информация не была убрана во время формирования и привела к появлению нескольких фанкциональных и нефункциональных требований, которые, в свою очередь, привели к появлению 2 сущностей в ER-диаграмме, попытке нейросети ввести дополнительные студии во время создания мокап-данных, а также ненужных тестов. 

2. Проектирование системы.
С целью проектирования и имплементации системы была выбрана платная версия ChatGPT + Codex. Был сконфигурирован файл AGENTS.md, который приписывал роль бизнес-аналитика нейронной сети, и регламентировал ее работу. В файл была добавлена структура проекта, формат логирования работы с нейросетью (файл .agents/AI_JOURNAL.md), скоуп МВП, правила работы, а также правила общения.
Отсутствие жесткого контроля за техническим заданием немедленно привело к упомянутым выше проблемам, что потребовало правки ТЗ на этапе, когда это делать запрещено. Часть работы по исправлению ошибок приведена уже в начале файла логов AI_JOURNAL.md.
В дальнейшем работа следовала примерам из Лекции 2. Нейронная сеть предложила стак Expo + React Native + TypeScript, TanStack Query, Zustand, React Hook Form и Zod, для бэкенда - mock-backend на Go 1.22 с chi, PostgreSQL, pgx, SQL-миграциями и Docker Compose. API описан контрактом OpenAPI 3.0.3.
Работа в Фигме над экранами не проводилась. С этого момента процесс разработки расходится с Лекцией 2.
На этапе ручного тестирования были выявлены (ожидаемые, целью 1 этапа была формировка ТЗ бесплатной нейронкой с минимальным хенд-холдингом) серьезные пробелы в UI\UX. С этого момента все баги и необходимые исправления по доведению приложения до МВП логировались в папке logs.
На данном этапе приложение достигло состояния МВП, которое было установлено на 1 этапе работы.

3. Тестирование.
По причине сжатых сроков на реальную работу по тестированию не хватило времени. Недостаточно жестко прописанное техническое задание на 1 этапе не позволило нейронной сети без серьезного "дирижирования" обеспечить достаточного покрытия тестами - результат этой работы скуден и находится в папке testing. Результат можно было бы серьезно улучшить, дав возможность провести углубленную правку технического задания, а также применив показанные на Лекции 3 шаблоны тестирования, но их разработка уже не укладывалась во временной промежуток.

4. Заключение.
Проработка ТЗ на первом этапе оказалась крайне недостаточной, и идея проверить, насколько отсутствие дирижирования на этом этапе скажется на конечном продукте, оказалась сугубо деструктивной. Можно резюмировать, что провал тестирования продиктован в большей мере не закрытыми дырами в техническом задании, нежели отсутствием профессиональных шаблонов, направляющих работу нейронки. Конечный результат, демонстрирующий, на глаз при поспешной проверке, корректную работу, скорее всего, будет подвержен багам и в текущем виде является демонстрационной заготовкой, а не реалистичным МВП. 
Однако, задание требовало получение результата в состоянии цейтнота и демонстрацией реалистичности подобного подхода, и в этом контексте полученный результат можно назвать крайне ограниченно, но успешным.

5. Запуск и проверка работоспособности.
Мокап-сервер запускается и содержит мокап-информацию о предстоящих классах, а также об одном клиенте с наполненной историей классов с целью тестирования.
Мобильное приложение требует android studio, запускается на эмуляторе устройства, функционал МВП реализован и находится в рабочем состоянии - позволяет записываться на классы одному \ группой, отменять заказы, оценивать шеф-повара, требует указания наличия аллергии при бронировании, позволяет при бронировании взять инвентарь впрокат.
