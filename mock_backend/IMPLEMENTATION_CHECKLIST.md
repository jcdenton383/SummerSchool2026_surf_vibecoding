# План-чеклист реализации mock-backend

## Назначение

Документ фиксирует пошаговый план реализации демонстрационного mock-backend для MVP приложения "Шеф-стол".

Mock-backend нужен для проверки клиентского приложения, API-контракта, бизнес-ограничений бронирования, PostgreSQL-схемы и передачи проекта другим пользователям.

## Границы

### Входит

- HTTP API по `api/openapi.yaml`.
- PostgreSQL-БД.
- Миграция инициализации схемы.
- Seed-данные для демонстрационных сценариев.
- Docker Compose для запуска mock-backend и PostgreSQL.
- Базовая bearer-auth имитация без production-безопасности.
- Бизнес-ограничения из ТЗ на уровне сервиса и частично на уровне БД.

### Не входит

- Production-бэкенд.
- Реальные платежи.
- Админка.
- Интерфейс шефа.
- Production SMS/push.
- Полноценная безопасность, RBAC, OAuth, refresh-token flow.
- Управление расписанием через API.
- Сущность, таблица, seed-данные или endpoint для адреса единственной студии; адрес на баннере является только статическим UI/UX-контентом.
- Реальная интеграция с платёжными, CRM, кассами или внешними сервисами.

## Источники истины

- `api/openapi.yaml` — endpoint-ы, схемы запросов/ответов, ошибки.
- `TZ_application/03-business-logic.md` — бизнес-правила.
- `TZ_application/04-api-contract-map.md` — связь экранов и API.
- `TZ_application/05-data-and-access.md` — сущности, доступы, инварианты.
- `design/ER-model-and-entity-models.md` — ER-модель.
- `TZ_application/06-acceptance-checklist.md` — приёмочные сценарии.

## Целевой стек

- Go.
- chi для HTTP-роутинга.
- pgx для PostgreSQL.
- golang-migrate для миграций.
- PostgreSQL.
- Docker Compose.
- Стандартный `testing` package для unit/integration tests.

## Итеративный план реализации

Итерации нужны, чтобы mock-backend собирался и проверялся постепенно. Каждая итерация должна завершаться рабочим состоянием, которое можно запустить, проверить и использовать как основу следующего шага.

### Итерация 0. Каркас и запуск

- [ ] Создать минимальный Go-сервис.
- [ ] Добавить конфигурацию через env.
- [ ] Подключить роутер и единый формат JSON-ответов.
- [ ] Реализовать `GET /health` без бизнес-логики.
- [ ] Подготовить базовый Dockerfile или команду локального запуска.
- [ ] Критерий выхода: сервис запускается локально и `GET /health` возвращает успешный ответ.

### Итерация 1. БД, миграции и демонстрационные данные

- [ ] Подключить PostgreSQL.
- [ ] Создать миграцию схемы для доменных таблиц MVP.

- [ ] Проверить, что seed покрывает happy path, отсутствие мест, отмену студией и завершённый класс.
- [ ] Критерий выхода: БД поднимается с чистой схемой, seed воспроизводим, таблица/сущность студии не создаётся.

- [ ] Критерий выхода: backend запускается одной командой, тесты проходят, README достаточен для передачи проекта.

## Этап 2. PostgreSQL-схема

### Таблицы

- [ ] `clients`.
- [ ] `class_programs`.
- [ ] `chefs`.
- [ ] `schedule_slots`.
- [ ] `bookings`.
- [ ] `booking_participants`.
- [ ] `chef_reviews`.
- [ ] `notifications`.
- [ ] `legal_terms`.

### Enum/check-ограничения

- [ ] `slot_status`: `available`, `full`, `cancelled_by_studio`, `completed`.
- [ ] `booking_status`: `confirmed`, `offline_payment_pending`, `cancelled_by_client`, `cancelled_by_studio`, `completed`.
- [ ] `payment_status`: `not_required`, `pending_offline`, `paid_offline`, `refunded`, `refund_pending`.
- [ ] `payment_method`: `offline`, `online_placeholder`.
- [ ] `payment_choice_status`: `selected`, `placeholder_shown`, `fallback_to_offline`.
- [ ] `participant_status`: `active`, `cancelled_by_client`, `cancelled_by_studio`.
- [ ] `allergy_status`: `none`, `has_allergy`.
- [ ] `equipment_option`: `own`, `rental`.
- [ ] `notification_type`: `class_reminder`, `studio_cancelled_class`.

### Обязательные ограничения БД

- [ ] Деньги хранить без `float`: `amount_minor integer` или `numeric(12,2)`.
- [ ] `capacity > 0`.
- [ ] `available_seats >= 0`.
- [ ] `available_seats <= capacity`.
- [ ] `duration_minutes > 0`.
- [ ] `seat_price_amount >= 0`.
- [ ] `rental_price_per_participant_amount >= 0`.
- [ ] `booking_participants.name` не пустой.
- [ ] `allergy_comment` обязателен при `allergy_status = 'has_allergy'`.
- [ ] `allergy_comment` допускается пустым при `allergy_status = 'none'`.
- [ ] `chef_reviews.rating between 1 and 5`.
- [ ] `unique (booking_id)` для `chef_reviews`, чтобы не было повторной оценки.
- [ ] Внешние ключи:
  - [ ] `bookings.client_id -> clients.id`;
  - [ ] `bookings.slot_id -> schedule_slots.id`;
  - [ ] `booking_participants.booking_id -> bookings.id`;
  - [ ] `schedule_slots.program_id -> class_programs.id`;
  - [ ] `schedule_slots.chef_id -> chefs.id`;
  - [ ] `chef_reviews.booking_id -> bookings.id`;
  - [ ] `chef_reviews.client_id -> clients.id`;
  - [ ] `chef_reviews.chef_id -> chefs.id`;
  - [ ] `notifications.client_id -> clients.id`;
  - [ ] `notifications.booking_id -> bookings.id`;
  - [ ] `notifications.slot_id -> schedule_slots.id`.

### Что не переносить только в БД

- [ ] Не пытаться одной миграцией полностью закрыть атомарное бронирование мест.
- [ ] Не пытаться одной миграцией полностью закрыть правило 24 часов.
- [ ] Не делать физическое удаление броней и участников основным сценарием.

- [ ] Не добавлять таблицы админов, ролей шефа и владельца.

## Риски

- Бронирование мест требует транзакций; без них demo может показывать некорректную доступность.
- Часть ограничений должна жить в сервисной логике, а не только в SQL.
- Слишком подробная имитация production-бэкенда раздует MVP.
- Нужно не добавлять реальные платежи и админку под видом удобства тестирования.
