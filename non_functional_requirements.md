# Нефункциональные требования

| ID | Формулировка | Приоритет | Источник |
|---|---|---|---|
| NFR-001 | Приложение должно быть кроссплатформенным и поддерживать Android и iOS в рамках единой кодовой базы. | Высокий | `requirements.md` §1-2 |
| NFR-002 | Интерфейс должен быть оптимизирован для использования на мобильном устройстве. | Высокий | `requirements.md` §1; `domainanalysis.md` §5 |
| NFR-003 | Клиентское приложение должно работать только в границах клиентского сценария и не включать админские или шефские интерфейсы. | Высокий | `requirements.md` §2-3; `domainanalysis.md` §6 |
| NFR-004 | Существующий бэкенд должен рассматриваться как источник истины по расписанию, слотам, вместимости, статусам и бронированиям. | Высокий | `requirements.md` §8; `domainanalysis.md` §3, §5 |
| NFR-005 | Канонической схемой данных для клиентского приложения должен быть API-контракт. | Высокий | `domainanalysis.md` §5 |
| NFR-006 | Приложение не должно требовать миграции исторических данных или легаси-схем. | Средний | `domainanalysis.md` §5 |
| NFR-007 | При ошибках API, плохом интернете или временной недоступности сервера приложение должно показывать клиенту понятное сообщение без технических деталей внутренней реализации. | Высокий | `requirements.md` §8, §10 |
| NFR-008 | Клиентские сообщения об ограничениях, ошибках бронирования, отмене и возврате должны быть понятными для пользователя. | Высокий | `requirements.md` §8-10 |
| NFR-009 | Приложение должно явно и заранее визуализировать критичные ограничения клиентского сценария: невозможность поздней отмены, отсутствие мест, отменённый слот и необходимость указать аллергии. | Высокий | `requirements.md` §4, §8-10 |
| NFR-010 | Персональные данные клиента, включая телефон, имена участников и сведения об аллергиях, должны обрабатываться только в объёме, необходимом для оформления и сопровождения брони. | Высокий | `requirements.md` §5, §10 |
| NFR-011 | Пользовательский интерфейс должен сохранять единые названия статусов брони во всех экранах приложения. | Средний | `requirements.md` §6, §11 |
