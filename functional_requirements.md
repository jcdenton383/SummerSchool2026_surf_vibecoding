# Функциональные требования

| ID | Формулировка | Приоритет | Источник |
|---|---|---|---|
| FR-001 | Система должна позволять клиенту авторизоваться по номеру телефона и SMS-коду или временному паролю. | Высокий | `requirements.md` §2, §4; `use_cases.md` UC-001 |
| FR-002 | Система должна автоматически создавать аккаунт клиента при первом успешном входе по номеру телефона. | Высокий | `requirements.md` §2, §4; `user_stories.md` US-001 |
| FR-003 | Система должна по умолчанию отображать клиенту список классов на ближайшие 7 дней от текущего момента. | Высокий | `brief/brief-cooking.md` R-027; `requirements.md` §2; `user_stories.md` US-002 |
| FR-004 | Система должна позволять клиенту выбрать другой диапазон дат для просмотра более длинного периода расписания. | Высокий | `brief/brief-cooking.md` R-027; `requirements.md` §2; `domainanalysis.md` §5 |
| FR-005 | Система не должна предоставлять в MVP поиск и маркетинговые фильтры по каталогу, кроме фильтра даты. | Средний | `requirements.md` §2; `domainanalysis.md` §6 |
| FR-005A | Если расписания на ближайшие 7 дней нет, система должна показывать empty state «Пока нет доступных классов». | Высокий | `brief/brief-cooking.md` R-027; `domainanalysis.md` §5 |
| FR-006 | Система должна отображать для класса название, направление, сложность, длительность, блюда, имя шефа, рейтинг шефа, дату и время. | Высокий | `requirements.md` §5; `use_cases.md` UC-002 |
| FR-007 | Система должна отображать необходимый инвентарь для каждого класса. | Высокий | `requirements.md` §5, §10 |
| FR-008 | Система должна отображать количество свободных мест и максимальную вместимость класса с учётом ограничений конкретного класса. | Высокий | `requirements.md` §5, §9 |
| FR-009 | Система должна отображать стоимость места, стоимость проката инвентаря и итоговую сумму до подтверждения записи. | Высокий | `requirements.md` §5-6; `user_stories.md` US-007 |
| FR-010 | Система должна позволять клиенту оформить запись одного участника на выбранный класс. | Высокий | `requirements.md` §4; `use_cases.md` UC-003 |
| FR-011 | Система должна позволять клиенту оформить групповую запись на выбранный класс. | Высокий | `requirements.md` §4; `use_cases.md` UC-004 |
| FR-012 | Система должна позволять клиенту добавить участников в существующую предстоящую запись. | Высокий | `requirements.md` §4; `use_cases.md` UC-005 |
| FR-013 | Система должна собирать имя основного клиента и номер телефона при оформлении записи. | Высокий | `requirements.md` §5 |
| FR-014 | Система должна собирать имя каждого дополнительного участника при групповой записи или добавлении участников. | Высокий | `requirements.md` §5 |
| FR-015 | Система должна требовать явный выбор «аллергии есть / нет» для каждого участника. | Высокий | `requirements.md` §5, §9 |
| FR-016 | Система должна запрещать оформление записи, если по любому участнику не указан статус аллергии. | Высокий | `requirements.md` §5, §9-10 |
| FR-017 | Система должна требовать текстовый комментарий, если по участнику выбрано «аллергии есть». | Высокий | `requirements.md` §5, §9 |
| FR-018 | Система должна сохранять данные об аллергиях на уровне конкретной брони и участника, а не как постоянные данные профиля клиента. | Высокий | `requirements.md` §5 |
| FR-019 | Система должна позволять выбрать собственный инвентарь или платный прокат для каждого участника. | Высокий | `requirements.md` §2, §4-6 |
| FR-020 | Система должна поддерживать выбор оффлайн-оплаты перед классом. | Высокий | `requirements.md` §6 |
| FR-021 | Система должна поддерживать выбор онлайн-оплаты как заглушки без проведения реального платежа. | Средний | `requirements.md` §6, §10 |
| FR-022 | При выборе онлайн-оплаты система должна показать заглушку и предложить перейти на оффлайн-оплату перед классом. | Средний | `requirements.md` §6 |
| FR-023 | Если клиент после онлайн-заглушки соглашается на оффлайн-оплату, система должна отправить запрос на создание брони в API. | Средний | `requirements.md` §6 |
| FR-024 | Если клиент после онлайн-заглушки отказывается от оффлайн-оплаты, система не должна создавать бронь. | Средний | `requirements.md` §6 |
| FR-025 | Система должна создавать или изменять бронь только после финального подтверждения API. | Высокий | `requirements.md` §8; `domainanalysis.md` §5 |
| FR-026 | Система должна отображать успешную запись в аккаунте клиента. | Высокий | `requirements.md` §4, §10 |
| FR-027 | Система должна позволять отменить всю запись не позднее чем за 24 часа до начала класса. | Высокий | `requirements.md` §4, §9 |
| FR-028 | Система должна позволять отменить отдельных участников групповой записи не позднее чем за 24 часа до начала класса. | Высокий | `requirements.md` §4, §9 |
| FR-029 | Система должна блокировать отмену записи, если до начала класса осталось менее 24 часов. | Высокий | `requirements.md` §4, §9-10 |
| FR-030 | Система должна показывать правило невозможности отмены менее чем за 24 часа при оформлении заказа и в карточке брони. | Высокий | `requirements.md` §4, §10 |
| FR-031 | При отмене записи или участников система должна отправлять запрос в API на освобождение мест. | Высокий | `requirements.md` §4, §8 |
| FR-032 | После отмены система должна отображать освобождённые места как доступные на основании обновлённых данных API. | Высокий | `requirements.md` §4, §9 |
| FR-033 | Система должна показывать заглушку возврата денег при отмене не позднее чем за 24 часа до начала класса. | Средний | `requirements.md` §4, §6 |
| FR-034 | Система должна отправлять push-напоминание о классе за 1 день до его начала. | Высокий | `requirements.md` §7 |
| FR-035 | Если в заказе класса не выбран прокат инвентаря, push-напоминание должно включать фразу «не забудьте инвентарь для класса» или близкую по смыслу. | Высокий | `requirements.md` §7, §10 |
| FR-036 | При отмене класса студией система должна отобразить статус отмены в приложении. | Высокий | `requirements.md` §4, §7 |
| FR-037 | При отмене класса студией система должна отправить клиенту push-уведомление и SMS. | Высокий | `requirements.md` §4, §7 |
| FR-038 | При отмене класса студией система должна показать заметную инструкцию по возврату денег. | Высокий | `requirements.md` §4, §10 |
| FR-039 | Система должна запрещать повторную запись на отменённый студией слот. | Высокий | `requirements.md` §8; `domainanalysis.md` §4 |
| FR-040 | Система должна позволять клиенту поставить шефу оценку от 1 до 5 звёзд после завершённого класса. | Средний | `requirements.md` §4, §10 |
| FR-041 | Система должна позволять клиенту оставить необязательный текстовый отзыв о шефе. | Средний | `requirements.md` §4, §10 |
| FR-042 | Система должна отображать рейтинг шефа в расширенной карточке класса. | Средний | `requirements.md` §4-5 |
| FR-043 | Система должна показывать условия отмены и политику обработки данных. | Высокий | `requirements.md` §2, §10 |
| FR-044 | Если на экране есть баннер с названием студии, адрес единственной студии допускается только как статический UI/UX-текст макета; система не должна получать, хранить или обновлять адрес через API/БД/seed. | Средний | `requirements.md` §2, §9-10 |
| FR-045 | Система должна обрабатывать отказ API из-за отсутствия мест, устаревших данных, отменённого слота и ошибок валидации. | Высокий | `requirements.md` §8, §10 |
| FR-046 | При отказе API система должна показать понятное сообщение, не создавать локально неподтверждённую бронь и обновить данные слота. | Высокий | `requirements.md` §8 |
| FR-047 | API-контракт должен предоставлять список классов и слотов на ближайшие 7 дней по умолчанию и за выбранный диапазон дат при использовании фильтра даты. | Высокий | `brief/brief-cooking.md` R-004, R-027; `domainanalysis.md` §5 |
| FR-048 | API-контракт должен предоставлять карточку класса с программой, меню, шефом, рейтингом шефа, датой, временем, длительностью, вместимостью, свободными местами, стоимостью места и тарифом проката. | Высокий | `requirements.md` §8; `domainanalysis.md` §5 |
| FR-049 | API-контракт должен поддерживать создание брони, групповой брони и добавление участников с передачей данных участников, аллергий, выбора инвентаря, способа оплаты и итоговой суммы. | Высокий | `requirements.md` §4-6, §8 |
| FR-050 | API-контракт должен поддерживать отмену всей брони и отдельных участников с освобождением мест при соблюдении правила 24 часов. | Высокий | `requirements.md` §4, §8-9 |
| FR-051 | API-контракт должен возвращать клиентские статусы брони, включая подтверждённую бронь, бронь с оффлайн-оплатой, отмену клиентом, отмену студией и завершённый класс. | Высокий | `requirements.md` §6, §8; `domainanalysis.md` §4 |
| FR-052 | API-контракт должен возвращать типизированные ошибки для отсутствия мест, устаревших данных, отменённого слота, запрета повторной записи, ошибок валидации и временной недоступности сервера. | Высокий | `requirements.md` §8 |
| FR-053 | API-контракт должен поддерживать сохранение оценки шефа от 1 до 5 и необязательного текстового отзыва после завершения класса. | Средний | `requirements.md` §4, §10 |
| FR-054 | Система не должна предоставлять функции лояльности, бонусов, скидок и статусов постоянных клиентов в MVP. | Средний | `requirements.md` §2 |
| FR-055 | Система не должна предоставлять функции управления расписанием, программами, шефами и административной отменой класса. | Высокий | `requirements.md` §2-3; `domainanalysis.md` §6 |
