# BUG-20260706-cancelled-booking-seat-release — Проверка освобождения места после отмены брони клиентом

## Статус

`fixed`

## Где найден

- Пользовательское свидетельство: после отмены клиентом брони место может не освобождаться в расписании.
- Экран отмены: `frontend/app/account/cancel.tsx`
- Backend отмены: `mock_backend/internal/service/service.go`, `mock_backend/internal/storage/postgres/store.go`

## Связь с ТЗ и требованиями

- `TZ_application/01-main-specification.md` — отмена брони клиентом.
- `TZ_application/02-screen-specification.md` — SCR-013.
- `TZ_application/03-business-logic.md` — изменение свободных мест при отмене.
- `api/openapi.yaml` — `POST /bookings/{bookingId}/cancel`, `POST /bookings/{bookingId}/participants/{participantId}/cancel`.

## Описание

Нужно проверить пользовательское наблюдение: при отмене клиентом брони его место не освобождается в расписании.

По текущему backend-коду место должно освобождаться:

- `CancelBooking` считает активных участников и вызывает `updateSlotSeats(ctx, tx, slotID, active)`;
- `CancelParticipant` вызывает `updateSlotSeats(ctx, tx, slotID, 1)`;
- `updateSlotSeats` увеличивает `available_seats` и переводит `full` обратно в `available`, если мест стало больше нуля.

При этом во frontend после успешной отмены инвалидируются только query брони и списка броней, но не инвалидируются query расписания/слота. Поэтому возможная причина пользовательского наблюдения — stale cache на экране расписания, а не ошибка пересчёта backend.

## Шаги воспроизведения

1. Авторизоваться.
2. Открыть слот в расписании и запомнить `availableSeats`.
3. Создать бронь на 1 участника.
4. Проверить, что `availableSeats` уменьшился.
5. Перейти в «Мои записи».
6. Отменить всю бронь или одного участника.
7. Вернуться в расписание.
8. Проверить, увеличился ли счётчик свободных мест.

## Фактический результат

По свидетельству пользователя: место может не освобождаться в расписании.

Факт не подтверждён запуском end-to-end проверки в текущем ходе работы.

## Ожидаемый результат

После успешной отмены:

- backend увеличивает `availableSeats` на количество отменённых активных участников;
- если слот был `full`, но появилось место, статус становится `available`;
- frontend после возврата в расписание показывает актуальное количество свободных мест.

## Причина

Предварительная гипотеза:

- backend-логика освобождения мест присутствует;
- frontend не инвалидирует `queryKeys.slots(...)` и `queryKeys.slot(slotId)` после отмены;
- пользователь может видеть устаревшие данные расписания из React Query cache.

## Исправление

Требуется проверить фактическое поведение через запуск приложения и/или backend API.

Если подтвердится stale cache:

- после `cancelBooking` и `cancelParticipant` инвалидировать query расписания и слота;
- при возврате в расписание выполнить refetch;
- проверить, что ответ API уже содержит актуальный `slot`, но UI использует его или обновляет кэш.

Если подтвердится backend-дефект:

- добавить тест на `CancelBooking` и `CancelParticipant`;
- проверить транзакцию и `available_seats`;
- исправить пересчёт мест.
