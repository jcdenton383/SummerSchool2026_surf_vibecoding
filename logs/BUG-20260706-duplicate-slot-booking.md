# BUG-20260706-duplicate-slot-booking — Повторная запись клиента на один и тот же класс

`fixed`

## Описание

Авторизованный пользователь может создать несколько отдельных броней на один и тот же слот. Это ломает модель MVP: если клиент уже записан на класс, повторный выбор этого класса должен вести не к новой брони, а к добавлению дополнительных участников в существующую группу.

## Шаги воспроизведения

1. Авторизоваться в мобильном приложении.
2. Открыть расписание.
3. Выбрать доступный класс.
4. Создать бронь на этот класс.
5. Вернуться в расписание.
6. Снова выбрать тот же класс.
7. Повторно пройти создание брони.

## Фактический результат

Пользователь может создать новую отдельную бронь на тот же класс.

По коду backend `Service.CreateBooking` и `Store.CreateBooking` проверяют статус слота и количество мест, но не проверяют активную бронь текущего клиента на тот же `slotId`.

## Ожидаемый результат

Если у пользователя уже есть активная бронь на слот:

- в расписании и карточке класса явно показать факт записи;
- при выборе карточки не запускать создание новой брони;
- предложить только сценарий добавления дополнительных людей в существующую группу;
- считать цену добавления только за новых участников и их прокат, без повторной оплаты за уже записанного клиента;
- backend должен запрещать создание второй активной брони на тот же `clientId + slotId`.

## Причина

Предварительно:

- frontend не связывает список слотов с `GET /bookings/my` и не знает, на какие слоты пользователь уже записан;
- `frontend/app/schedule/[slotId].tsx` всегда ведёт в `/booking/{slotId}`, если слот доступен;
- backend не имеет проверки существующей активной брони перед `insert into bookings`.

## Исправление

Требуется:

- на backend добавить проверку активной брони клиента на слот перед созданием новой брони;
- вернуть управляемую ошибку, например `already_exists` или отдельный код, согласованный с API;
- на frontend при авторизованной сессии получать мои активные брони и сопоставлять их со слотами;
- для уже записанного слота вести пользователя в `/account/add-participants?bookingId=...`, а не в создание новой брони;
- в UI явно показать текст вроде `Вы уже записаны на этот класс`.
