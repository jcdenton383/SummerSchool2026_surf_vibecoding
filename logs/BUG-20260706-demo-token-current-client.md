# BUG-20260706-demo-token-current-client — Защищённые методы mock-backend работали не с текущим клиентом

## Статус

`fixed`

## Где найден

- Автоматизированное тестирование с помощью ИИ без участия разработчика: осмотр архитектуры и поведения системы, не unit-тесты.
- `mock_backend/internal/service/service.go`
- `mock_backend/internal/http/router.go`
- `mock_backend/internal/storage/postgres/store.go`

## Связь с ТЗ и требованиями

- `TZ_application/05-data-and-access.md` — клиент должен работать только со своими бронями.
- `TZ_application/01-main-specification.md` — авторизованный клиент просматривает и изменяет только свои записи.
- `api/openapi.yaml` — защищённые endpoints должны относиться к current client.

## Описание

После проверки bearer-token backend получал текущего клиента через `DemoClient()`, который возвращал первого клиента из БД. При наличии нескольких клиентов защищённые методы могли читать и изменять данные не того пользователя.

## Шаги воспроизведения

1. Авторизоваться с телефоном A.
2. Авторизоваться с телефоном B.
3. С токеном B вызвать `/me`, `/bookings/my` или `POST /bookings`.

## Фактический результат

Backend мог использовать первого клиента из таблицы `clients`, а не клиента, для которого был выдан токен.

## Ожидаемый результат

Защищённые методы должны работать с клиентом, связанным с bearer-token.

## Причина

Demo-auth использовал один статический токен и не связывал его с `clientId`.

## Исправление

Demo-token теперь формируется как `<demoToken>:<clientId>`. Авторизация извлекает `clientId` из токена и загружает клиента через `GetClientByID`.

## Изменённые артефакты

- `mock_backend/internal/service/service.go` — токен привязан к `clientId`, `CurrentClient` принимает token.
- `mock_backend/internal/http/router.go` — auth middleware передаёт token в service.
- `mock_backend/internal/storage/postgres/store.go` — удалён небезопасный `DemoClient`.
- `logs/BUG-20260706-demo-token-current-client.md` — создан баг-лог.
