# BUG-20260706-session-restore-client — Frontend не восстанавливал клиента после перезапуска

## Статус

`fixed`

## Где найден

- Автоматизированное тестирование с помощью ИИ без участия разработчика: осмотр архитектуры и поведения системы, не unit-тесты.
- `frontend/src/features/auth/sessionStore.ts`
- `frontend/app/account/index.tsx`

## Связь с ТЗ и требованиями

- `TZ_application/01-main-specification.md` — клиент должен видеть свои записи и данные аккаунта после авторизации.
- `TZ_application/05-data-and-access.md` — клиентские данные приходят из API, а не подменяются локальной заглушкой.
- `api/openapi.yaml` — `GET /me` возвращает current client.

## Описание

При восстановлении приложения из локального хранилища frontend восстанавливал только `accessToken`. Объект `client` оставался `null`, поэтому в аккаунте показывался fallback `demo-session`.

## Шаги воспроизведения

1. Войти в приложение.
2. Перезапустить или обновить приложение.
3. Открыть экран «Мои записи».

## Фактический результат

Пользователь считается авторизованным по токену, но данные клиента не восстановлены.

## Ожидаемый результат

После восстановления токена приложение должно получить текущего клиента через `/me` или очистить сессию, если токен недействителен.

## Причина

`restore()` читал token из `tokenStorage`, но не вызывал `api.getCurrentClient()`.

## Исправление

`restore()` теперь:

- очищает сессию, если токена нет;
- временно устанавливает token;
- вызывает `api.getCurrentClient()`;
- сохраняет `client`;
- очищает сессию при ошибке восстановления.

## Изменённые артефакты

- `frontend/src/features/auth/sessionStore.ts` — добавлено восстановление клиента через `/me`.
- `logs/BUG-20260706-session-restore-client.md` — создан баг-лог.
- `.agents/AI_JOURNAL.md` — добавлена запись о багфиксе.
