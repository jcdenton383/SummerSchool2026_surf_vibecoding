# BUG-20260706-schedule-null-items-crash — Падение расписания при пустом коротком периоде

## Статус

`fixed`

## Где найден

- Экран `SCR-003` / `frontend/app/schedule/index.tsx`
- Сценарий выбора периода расписания на 0 или 1 день.

## Связь с ТЗ и требованиями

- `screen_registry.md` — SCR-003 Расписание классов, SCR-004 Фильтр даты.
- `design/1.1-calendar-date-filter-feature.md` — календарный выбор дат.
- `api/openapi.yaml` — `GET /slots?from={date}&to={date}`.

## Описание

При выборе короткого периода расписания приложение падало с ошибкой:

`TypeError: Cannot read property 'length' of null`

Причина: backend при пустой выборке мог сериализовать nil-slice как `items: null`, а экран расписания обращался к `query.data?.items.length`.

## Шаги воспроизведения

1. Открыть SCR-003.
2. Выбрать период расписания с расстоянием 0 или 1 день, в котором нет классов.
3. Дождаться ответа API.

## Фактический результат

Приложение падает на экране расписания.

## Ожидаемый результат

Экран не падает и показывает empty state: `На выбранные даты доступных классов нет.`

## Причина

- В `mock_backend/internal/storage/postgres/store.go` пустые списки создавались как nil-slice.
- В `frontend/app/schedule/index.tsx` не было локальной нормализации `items`.

## Исправление

- Backend теперь инициализирует пустые списки как `[]`, а не `null`.
- API-клиент frontend нормализует `items: null` в `items: []` для списковых ответов.
- Экран расписания использует локальный `slots = query.data?.items ?? []`.
- Добавлен регрессионный тест на нормализацию `null` в пустой массив.

## Изменённые артефакты

- `mock_backend/internal/storage/postgres/store.go` — пустые списки слотов, броней и уведомлений инициализируются как пустые массивы.
- `frontend/src/api/client.ts` — добавлена нормализация списковых ответов.
- `frontend/app/schedule/index.tsx` — добавлена защита от отсутствующего списка.
- `frontend/tests/api-client.test.ts` — добавлен регрессионный тест.
- `logs/BUG-20260706-schedule-null-items-crash.md` — создан баг-репорт.
