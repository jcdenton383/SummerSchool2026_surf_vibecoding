# Технологический стек MVP

## 1. Решение

Для реализации клиентского мобильного приложения "Шеф-стол" утверждается стек:

| Зона | Технология | Назначение |
|---|---|---|
| Базовая платформа | Expo | Быстрый запуск и сборка мобильного приложения для Android и iOS. |
| UI/runtime | React Native | Кроссплатформенный мобильный интерфейс. |
| Язык | TypeScript | Типизация экранов, API-моделей, форм и бизнес-логики. |
| Роутинг | Expo Router | Навигация между экранами приложения. |
| Серверное состояние | TanStack Query | Запросы к API, кэш, повторные попытки, состояния загрузки и ошибок. |
| Локальное состояние | Zustand | Сессия, черновики форм, выбранные фильтры, UI-состояния. |
| Формы | React Hook Form | Управление формами бронирования, авторизации и отзывов. |
| Валидация | Zod | Схемы валидации пользовательского ввода. |
| Защищённое хранение | Expo SecureStore | Хранение access token. |
| Моки API | Локальный mock-слой | Имитация бэкенда по `api/openapi.yaml` до подключения реального API. |
| Демонстрационный mock-backend | Go + chi | Отдельный HTTP-сервис для демонстрации API-контракта, бизнес-правил бронирования и сценарных ошибок. |
| База данных mock-backend | PostgreSQL | Инициализация схемы, хранение демонстрационных данных, проверка базовых ограничений модели. |
| Миграции БД | golang-migrate | Версионирование схемы PostgreSQL и seed-данных для демонстрационного запуска. |
| Контейнеризация | Docker Compose | Быстрый запуск мобильного приложения, mock-backend и PostgreSQL на другой машине. |
| Уведомления MVP | In-app simulator | Имитация SMS/push-уведомлений внутри приложения. |

## 2. Обоснование

Стек выбран для максимально быстрого создания MVP в режиме вайбкодинга:

1. React Native + Expo не требует нативной разработки отдельно под Android и iOS.
2. TypeScript и React-компоненты хорошо подходят для итеративной генерации и правки кода.
3. Expo снижает стоимость старта проекта, запуска на устройстве и подготовки тестовых билдов.
4. TanStack Query и Zustand покрывают основные сценарии состояния без сложной архитектуры.
5. React Hook Form и Zod позволяют быстро реализовать формы с проверками из ТЗ.
6. Mock-слой позволяет показать приложение без готового бэкенда, который находится вне скоупа задачи.
7. Демонстрационный mock-backend нужен для проверки контрактов, транзакционных сценариев бронирования и передачи проекта другим пользователям без доступа к реальному API.
8. PostgreSQL используется для проверки ограничений модели и воспроизводимого заполнения демонстрационными данными, но не превращает MVP в production-сервер.

## 4. Моковый режим и демонстрационный backend

До подключения реального API приложение должно работать в моковом режиме:

1. Расписание, карточки классов, брони, уведомления и отзывы берутся из локальных моков.
2. Моки должны поддерживать успешные сценарии и основные ошибки из ТЗ: `no_seats`, `stale_slot_data`, `slot_cancelled_or_unavailable`, `cancellation_window_closed`, `temporary_unavailable`.
3. Имитация SMS/push реализуется как внутренний экран или панель уведомлений.
4. Локально созданная бронь считается демонстрационной, но UI должен соблюдать правило ТЗ: подтверждённое состояние показывается только после успешного ответа mock API.
5. Для демонстрации связной системы допускается отдельный mock-backend, реализующий endpoint-ы из `api/openapi.yaml`.
6. Mock-backend должен использовать PostgreSQL-миграции для создания схемы БД и seed-данные для расписания, программ, шефов, клиентов, броней, уведомлений и отзывов.
7. Mock-backend обязан имитировать ключевые бизнес-ограничения: свободные места, статусы слотов и броней, обязательность аллергий, расчёт суммы, отмену за 24 часа и запрет повторной записи на отменённый студией слот.
8. Требования production-безопасности, реальные платежи, админка и реальные SMS/push не входят в mock-backend.

## 5. Что не берём в MVP

В рамках MVP не используются:

1. Нативная разработка на Swift/Kotlin.
2. Production-серверная часть; демонстрационный mock-backend допускается только для проверки MVP и передачи проекта.
3. Реальная онлайн-оплата.
4. Реальная отправка SMS.
5. Production push-инфраструктура.
6. Сложная дизайн-система до утверждения визуального стиля.
7. Сложные архитектурные паттерны сверх модульной структуры по фичам.
