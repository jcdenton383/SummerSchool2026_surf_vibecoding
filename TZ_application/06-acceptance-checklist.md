# Критерии приёмки MVP

## 1. Общие критерии

| ID | Критерий | Приоритет |
|---|---|---|
| AC-G01 | Приложение запускается на Android и iOS. | P0 |
| AC-G02 | В приложении нет админских, шефских и CRM-сценариев. | P0 |
| AC-G03 | Все бизнес-данные загружаются из API. Статический адрес на баннере, если он есть в макете, не считается бизнес-данными приложения. | P0 |
| AC-G04 | При ошибке API бронь не создаётся локально как подтверждённая. | P0 |
| AC-G05 | Все сообщения об ошибках и ограничениях понятны клиенту. | P1 |

## 2. Авторизация

| ID | Критерий | Приоритет |
|---|---|---|
| AC-A01 | Пользователь может запросить код по валидному телефону. | P0 |
| AC-A02 | Невалидный телефон не отправляется в API. | P0 |
| AC-A03 | Пользователь может подтвердить код и получить сессию. | P0 |
| AC-A04 | Неверный или истёкший код показывает ошибку без входа. | P0 |
| AC-A05 | Rate limit отображается с понятным ожиданием. | P1 |

## 3. Расписание и карточка класса

| ID | Критерий | Приоритет |
|---|---|---|
| AC-S01 | По умолчанию показываются классы на ближайшие 7 дней. | P0 |
| AC-S02 | Фильтр даты обновляет список по выбранному диапазону. | P0 |
| AC-S03 | При пустом расписании отображается "Пока нет доступных классов". | P1 |
| AC-S04 | Карточка класса показывает программу, меню, шефа, рейтинг, цену, места и инвентарь. | P0 |
| AC-S05 | Слот без мест или отменённый студией нельзя забронировать. | P0 |
| AC-S06 | Баннер студии может содержать статический UI/UX-текст названия и адреса; это не проверка API, БД, seed-данных или mock-backend. | P2 |

## 4. Создание брони

| ID | Критерий | Приоритет |
|---|---|---|
| AC-B01 | Клиент может оформить запись на одного участника. | P0 |
| AC-B02 | Клиент может оформить групповую запись. | P0 |
| AC-B03 | Нельзя продолжить без имени обязательного участника. | P0 |
| AC-B04 | Нельзя продолжить без выбора аллергии по каждому участнику. | P0 |
| AC-B05 | При выборе "аллергии есть" комментарий обязателен. | P0 |
| AC-B06 | Нельзя добавить участников больше, чем доступно мест. | P0 |
| AC-B07 | Итоговая сумма учитывает цену мест и прокат. | P0 |
| AC-B08 | Оффлайн-оплата создаёт бронь через `POST /bookings`. | P0 |
| AC-B09 | Онлайн-оплата показывает заглушку и не проводит платёж. | P1 |
| AC-B10 | Отказ от оффлайн-оплаты после онлайн-заглушки не создаёт бронь. | P0 |
| AC-B11 | При `201` пользователь видит экран успешной записи. | P0 |
| AC-B12 | Успешная запись появляется в аккаунте. | P0 |

## 12. Smoke-сценарии

1. Новый клиент входит по телефону, видит расписание, открывает карточку класса, оформляет одиночную бронь с оффлайн-оплатой и видит её в аккаунте.
2. Клиент оформляет групповую бронь, указывает аллергии и прокат по одному участнику, видит корректную итоговую сумму.
3. Клиент выбирает онлайн-оплату, видит заглушку, отказывается от оффлайн-оплаты, бронь не создаётся.
4. Клиент добавляет участника в существующую бронь, API подтверждает изменение, детали брони обновляются.
5. Клиент пытается отменить бронь менее чем за 24 часа, действие заблокировано.
6. API возвращает `no_seats` при создании, приложение не создаёт бронь и обновляет слот.
7. Студия отменяет класс, клиент видит статус отмены и инструкцию по возврату.
8. После завершённого класса клиент оставляет оценку шефу.
