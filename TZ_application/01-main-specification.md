# Полное техническое задание

**Проект:** клиентское мобильное приложение "Шеф-стол"  
**Версия ТЗ:** 1.0  
**Дата:** 2026-07-05  
**Платформы:** Android, iOS  
**Источник API:** `api/openapi.yaml`

## 1. Назначение

MVP должен позволить клиентам студии "Шеф-стол" самостоятельно просматривать расписание кулинарных классов, выбирать класс, оформлять одиночную или групповую запись, указывать участников, аллергии и инвентарь, выбирать оффлайн-оплату, видеть свои записи, отменять записи в разрешённый срок и оставлять оценку шефу после завершения класса.

Приложение работает поверх существующего бэкенда. Источник истины для расписания, свободных мест, статусов, броней, отмен и уведомлений - API. Клиентское приложение не создаёт неподтверждённые бизнес-сущности локально.

## 2. Состав MVP

В MVP входят:

1. Авторизация по телефону и коду подтверждения.
2. Автоматическое создание аккаунта при первом успешном входе.
3. Просмотр расписания на ближайшие 7 дней по умолчанию.
4. Фильтр расписания по диапазону дат.
5. Просмотр карточки класса с программой, меню, шефом, рейтингом, ценой, местами и инвентарём.
6. Оформление одиночной и групповой брони.
7. Сбор имени клиента и участников.
8. Обязательный выбор статуса аллергии по каждому участнику.
9. Обязательный комментарий при наличии аллергии.
10. Выбор собственного инвентаря или проката по каждому участнику.
11. Расчёт предварительной и итоговой суммы.
12. Оффлайн-оплата перед классом.
13. Заглушка онлайн-оплаты с переходом на оффлайн-оплату.
14. Просмотр предстоящих, завершённых и отменённых записей.
15. Добавление участников в существующую предстоящую бронь.
16. Отмена всей брони или отдельных участников не позднее чем за 24 часа до класса.
17. Отображение отмены класса студией.
18. Push-напоминание за 1 день до класса.
19. Push и SMS при отмене класса студией.
20. Оценка шефа от 1 до 5 и необязательный текстовый отзыв.
21. Экран условий отмены и политики обработки персональных данных.
22. Статический UI/UX-баннер с названием и адресом единственной студии как часть макета, без API, БД и бизнес-логики.

## 3. Не входит в MVP

1. Административные интерфейсы.
2. Интерфейс шефа.
3. Управление расписанием, программами, шефами и вместимостью.
4. Реальная онлайн-оплата и автоматический возврат денег.
5. Лояльность, бонусы, скидки и статусы постоянных клиентов.
6. Поиск и маркетинговые фильтры, кроме фильтра даты.
7. Email-уведомления и мессенджеры.
8. Хранение аллергий в постоянном профиле клиента.

## 4. Роли

| Роль | Возможности |
|---|---|
| Гость | Просмотр расписания, карточки класса, условий и политики данных. Для бронирования должен авторизоваться. |
| Авторизованный клиент | Создание и изменение только своих броней, просмотр своих записей, отмена в разрешённый срок, оценка шефа. |
| Бэкенд | Источник истины для расписания, доступности, статусов, проверок прав, уведомлений и бизнес-валидации. |
| Администратор / владелец | Не входит в приложение; управляет расписанием во внешней системе. |
| Шеф | Не входит в приложение; отображается как сущность класса и получает оценки. |

## 5. Основные пользовательские сценарии

### 5.1 Вход

1. Клиент вводит номер телефона.
2. Приложение отправляет `POST /auth/phone-code`.
3. Клиент вводит код.
4. Приложение отправляет `POST /auth/verify-code`.
5. При успехе сохраняется сессия, открывается расписание или исходный защищённый экран.

### 5.2 Просмотр и выбор класса

1. Приложение запрашивает `GET /slots`.
2. По умолчанию показываются ближайшие 7 дней.
3. Клиент может открыть фильтр даты и запросить другой период.
4. При выборе класса открывается карточка через `GET /slots/{slotId}`.

### 5.3 Создание брони

1. Клиент выбирает доступный класс.
2. Заполняет участников, аллергии и инвентарь.
3. Проверяет сумму и способ оплаты.
4. При оффлайн-оплате приложение отправляет `POST /bookings`.
5. При онлайн-оплате показывает заглушку; бронь создаётся только после согласия на оффлайн-оплату.
6. При успехе отображается экран успешной записи и бронь появляется в аккаунте.

### 5.4 Добавление участников

1. Клиент открывает предстоящую бронь.
2. Выбирает "Добавить участников".
3. Заполняет данные новых участников.
4. Приложение отправляет `POST /bookings/{bookingId}/participants`.
5. При успехе показывает обновлённую бронь и сумму.

### 5.5 Отмена

1. Клиент открывает детали брони.
2. Если до класса меньше 24 часов, отмена заблокирована.
3. Если отмена разрешена, клиент выбирает всю бронь или отдельных участников.
4. Приложение отправляет `POST /bookings/{bookingId}/cancel` или `POST /bookings/{bookingId}/participants/{participantId}/cancel`.
5. При успехе показывает обновлённый статус и заглушку возврата.

### 5.6 Отмена класса студией

1. Студия отменяет класс во внешней системе.
2. Бэкенд меняет статус слота и броней.
3. Клиент получает push и SMS.
4. Приложение показывает статус `cancelled_by_studio`, причину при наличии и инструкцию по возврату.
5. Повторная запись на этот слот запрещена.

### 5.7 Оценка шефа

1. После завершения класса бронь получает статус `completed`.
2. В аккаунте становится доступно действие "Оценить шефа".
3. Клиент выбирает 1-5 звёзд и может оставить текст.
4. Приложение отправляет `POST /chef-reviews`.

## 9. Общие правила UI

1. Основная навигация: расписание и аккаунт.
2. Все действия с бронью открываются из контекста класса или брони.
3. На экранах со студийным баннером адрес единственной студии допускается только как статический UI/UX-текст макета; он не является данными приложения, API, БД, seed или acceptance backend.
4. Ошибки форм показывать у полей, ошибки API - через понятное сообщение и возможность повторить действие.
5. Состояния загрузки не должны позволять повторно отправить один и тот же критичный запрос.
6. При любых конфликтах мест обновлять данные слота.
7. Терминология статусов должна быть единой:
   - подтверждена;
   - оффлайн-оплата перед классом;
   - отменена клиентом;
   - отменена студией;
   - завершена.

## 10. Интеграционные ограничения

1. Приложение не рассчитывает финальную доступность мест самостоятельно.
2. Предварительная сумма на клиенте нужна для прозрачности, но финальная сумма подтверждается API.
3. При ошибке `409`, `410` или `503` бронь не считается созданной.
4. При отмене или добавлении участников интерфейс обновляется только после успешного ответа API.
5. При истечении сессии защищённый экран переводит пользователя в авторизацию с возвратом к исходному сценарию после входа.
