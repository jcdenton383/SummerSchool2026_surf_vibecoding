# Use Cases

## UC-001. Регистрация и вход клиента

| Поле | Значение |
|---|---|
| ID | UC-001 |
| Приоритет | Высокий |
| Актор | Клиент |
| Формулировка | Клиент входит в приложение по номеру телефона и SMS-коду; система создаёт новый аккаунт или открывает существующий. |
| Предусловия | Приложение установлено; клиент имеет номер телефона. |
| Основной сценарий | 1. Клиент вводит телефон. 2. Система отправляет SMS-код или временный пароль. 3. Клиент вводит код. 4. Система авторизует клиента. 5. Система создаёт аккаунт, если он отсутствует. |
| Альтернативы/исключения | При ошибочном коде система показывает ошибку валидации. При плохом интернете система показывает сообщение о проблеме соединения. |
| Результат | Клиент авторизован и видит свой аккаунт. |
| Источник | `requirements.md` §4, §10 |

## UC-002. Просмотр класса

| Поле | Значение |
|---|---|
| ID | UC-002 |
| Приоритет | Высокий |
| Актор | Клиент |
| Формулировка | Клиент просматривает список классов и расширенную карточку выбранного класса. |
| Предусловия | API отдаёт классы и свободные места. |
| Основной сценарий | 1. Клиент открывает приложение. 2. Система показывает список классов. 3. Клиент выбирает класс. 4. Система показывает название, длительность, сложность, блюда, шефа, инвентарь, свободные места и стоимость. |

## UC-003. Запись одного участника

| Поле | Значение |
|---|---|
| ID | UC-003 |
| Приоритет | Высокий |
| Актор | Клиент |
| Формулировка | Клиент оформляет запись одного участника на выбранный класс. |
| Предусловия | Клиент авторизован; выбран класс со свободными местами. |
| Основной сценарий | 1. Клиент выбирает запись одного участника. 2. Указывает имя и телефон. 3. Явно выбирает наличие/отсутствие аллергии. 4. При наличии аллергии вводит комментарий. 5. Выбирает собственный инвентарь или прокат. 6. Видит итоговую сумму. 7. Выбирает способ оплаты. 8. Система отправляет запрос в API. 9. API подтверждает создание брони. |
| Альтернативы/исключения | Если аллергия не отмечена, запись невозможна. Если API отказал из-за отсутствия мест или устаревших данных, система показывает ошибку и обновляет слот. |
| Результат | Бронь одного места создана и отображается в аккаунте. |
| Источник | `requirements.md` §4-6, §8-10 |

## UC-004. Запись группы

| Поле | Значение |
|---|---|
| ID | UC-004 |
| Приоритет | Высокий |
| Актор | Клиент |
| Формулировка | Клиент оформляет групповую запись на класс. |
| Предусловия | Клиент авторизован; выбран класс. |
| Основной сценарий | 1. Клиент добавляет участников. 2. Для каждого участника указывает имя. 3. Для каждого участника выбирает наличие/отсутствие аллергии. 4. При наличии аллергии вводит комментарий. 5. Для каждого участника выбирает инвентарь. 6. Система показывает итоговую сумму. 7. Клиент выбирает оплату. 8. Система проверяет через API доступность нужного количества мест. |
| Альтернативы/исключения | Если свободных мест меньше, чем участников, система сообщает о невозможности записи и обновляет данные слота. |
| Результат | Групповая бронь создана либо клиент получил понятный отказ. |
| Источник | `requirements.md` §4-6, §8 |

## UC-005. Добавление участников в существующую запись

| Поле | Значение |
|---|---|
| ID | UC-005 |
| Приоритет | Высокий |
| Актор | Клиент |
| Формулировка | Клиент добавляет N участников в уже созданную запись. |
| Предусловия | У клиента есть предстоящая запись; до класса есть свободные места. |
| Основной сценарий | 1. Клиент открывает аккаунт. 2. Выбирает предстоящий класс. 3. Добавляет N участников. 4. Указывает их данные, аллергии и инвентарь. 5. Видит доплату. 6. Выбирает метод оплаты. 7. Система отправляет изменение в API. |
| Альтернативы/исключения | Если мест не хватает, система сообщает, что добавить указанное количество участников нельзя. |
| Результат | Участники добавлены в запись, свободные места уменьшены. |
| Источник | `requirements.md` §4, §8, §10 |

## UC-006. Отмена записи или участников

| Поле | Значение |
|---|---|
| ID | UC-006 |
| Приоритет | Высокий |
| Актор | Клиент |
| Формулировка | Клиент отменяет всю запись или отдельных участников не позднее чем за 24 часа до класса. |
| Предусловия | У клиента есть предстоящая запись; до класса осталось не менее 24 часов. |
| Основной сценарий | 1. Клиент открывает аккаунт. 2. Выбирает предстоящий класс. 3. Выбирает всю бронь или M участников. 4. Система показывает заглушку возврата. 5. Клиент подтверждает отмену. 6. Система отправляет запрос в API. |
| Альтернативы/исключения | Если до класса осталось менее 24 часов, система блокирует отмену и показывает сообщение о запрете. |
| Результат | Бронь или выбранные участники отменены, места освобождены. |
| Источник | `requirements.md` §4, §9-10 |

## UC-007. Обработка форс-мажорной отмены класса

| Поле | Значение |
|---|---|
| ID | UC-007 |
| Приоритет | Высокий |
| Актор | Существующий бэкенд, Клиент |
| Формулировка | Клиентское приложение обрабатывает статус отмены класса, полученный из внешней инфраструктуры. |
| Предусловия | Класс отменён во внешней системе; бэкенд передал статус отмены. |
| Основной сценарий | 1. Приложение получает статус отмены. 2. Клиент получает push. 3. Клиент получает SMS. 4. В приложении отображается статус отмены. 5. Приложение показывает заметную инструкцию по возврату денег. |
| Альтернативы/исключения | Повторная запись на отменённый студией слот запрещена. |
| Результат | Клиент уведомлён об отмене и видит инструкцию по возврату. |
| Источник | `requirements.md` §4, §7-8 |

## UC-008. Оценка шефа

| Поле | Значение |
|---|---|
| ID | UC-008 |
| Приоритет | Средний |
| Актор | Клиент |
| Формулировка | Клиент оценивает шефа после завершённого класса. |

| Альтернативы/исключения | Текстовый отзыв необязателен. |
